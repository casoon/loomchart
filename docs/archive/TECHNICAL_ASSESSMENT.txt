RUST/WASM SETUP: ⭐⭐⭐⭐ (4/5)
--------------------------------------
Stärken:
+ Saubere Workspace-Struktur mit klaren Verantwortlichkeiten
+ wasm-bindgen korrekt eingesetzt
+ Thread-local State Management (kein unsicheres Global State)
+ console_log + console_error_panic_hook für Debugging
+ Separate WASM-Builds (trading_ui vs chartcore)

Schwächen:
- Zwei WASM Module erhöhen Komplexität (Bundle Size, Init-Zeit)
- State zwischen WASM und JS teils dupliziert (z.B. Candles in JS + WASM)
- Boundary-Calls könnten optimiert werden (JSON serialization overhead)
- Memory Management: Keine SharedArrayBuffer-Nutzung für Zero-Copy

Risiko: MITTEL
Die Architektur ist solide, aber die Grenzfläche WASM↔JS ist ein Performance-
Bottleneck bei vielen Events (Mousemove, Wheel). Aktuell wird pro Event JSON
serialisiert - das skaliert nicht für Drawings mit 1000+ Objekten.


CHART-ENGINE (chartcore vs chart-core): ⭐⭐⭐ (3/5)
----------------------------------------------------
Problem: DUALITÄT

Es existieren ZWEI Chart-Engines:
1. chartcore (Rust/WASM) - crates/chartcore
   - Rendering System (Canvas2D via wasm-bindgen)
   - Chart State, Viewport, Buffer
   - Plugin System für Indikatoren

2. chart-core (TypeScript) - packages/chart-core
   - Eigenständige Chart-Implementierung
   - Canvas2D + WebGPU Renderer
   - ViewportManager, CandleSeries
   - Adaptive FPS, Invalidation System

Aktueller Zustand:
- trading_ui (WASM) benutzt chartcore als Bibliothek
- Frontend benutzt chart-core (TS)
- chart-bridge.ts verbindet WASM → TS Chart (window.loomChart)

Architektur-Frage:
Warum zwei Engines? Vermutlich:
- chartcore war der Plan (Rust-first)
- chart-core wurde parallel entwickelt (schnellere Iteration in TS)
- Jetzt koexistieren beide

Bewertung:
+ Beide funktionieren isoliert gut
+ chartcore hat bessere Plugin-Architektur
+ chart-core hat bessere Browser-Integration (Events, Performance)

- Doppelte Arbeit
- Unklar, welche Engine "Source of Truth" ist
- State-Sync-Problem zwischen beiden

Risiko: HOCH
Diese Dualität muss aufgelöst werden. Entweder:
A) chartcore wird primär, chart-core wird Wrapper
B) chart-core wird primär, chartcore nur für Indikatoren
C) Fusion beider Ansätze

Empfehlung: Option B (pragmatisch) oder C (langfristig besser)

4. DIE GROSSEN HERAUSFORDERUNGEN
================================================================================

1. DRAWING TOOLS
----------------
Warum so schwer?

Problem A: HIT-TESTING
- Bei 100 Trendlines: Welche ist bei (x,y) unter der Maus?
- Naive Lösung: Prüfe alle → O(n), zu langsam
- Richtige Lösung: Spatial Index (Quadtree/R-Tree)
- Rust-seitig: rstar crate
- Problem: Hit-Testing braucht Geometrie-Berechnungen
  * Punkt-zu-Linie-Distanz
  * Punkt-in-Polygon
  * Bounding-Box-Checks
- WASM↔JS Boundary: Wer macht Hit-Testing?
  * WASM: Schneller, aber Geometrie-Berechnungen per Event → Latenz
  * JS: Direkter DOM-Zugriff, aber langsamer bei 1000+ Objekten

Problem B: HANDLES & INTERACTION
- Jede Drawing hat N Handles (z.B. Trendline: 2, Fib: 2, Pitchfork: 4)
- Handle States: Idle, Hover, Dragging
- Snapping: Handle "magnetisch" zu OHLC, Grid, anderen Drawings
- Constraints: Horizontal-only, Vertical-only (Shift-Lock)
- Multi-Selection: Drag-Rectangle, Shift+Click
- Transformation: Move, Resize, Rotate

Code-Komplexität:
- State Machine: Drawing × Handle × InteractionState
- Event Handlers: MouseDown/Move/Up/Leave für jeden State
- Coordinate Transforms: Screen ↔ Data Space
- Undo/Redo: Jede Aktion = Command

Implementierung:
Option A (WASM-first):
+ Logic in Rust → testbar, deterministisch
+ State konsistent
- Event-Boundary-Latenz
- Debugging schwieriger

Option B (JS-first):
+ DOM-nah, schneller Feedback
+ Einfacher zu debuggen
- State-Drift möglich
- Doppelte Implementierung (WASM muss für Backtesting auch wissen)

Option C (Hybrid):
+ WASM hält Truth (Drawing State)
+ JS macht Hit-Testing + Event-Handling
+ JS ruft WASM-Commands (MoveHandle, etc.)
- Komplexe API-Grenze

Empfehlung: Option C mit gutem Command-Pattern

Problem C: SNAPPING/MAGNETISMUS
TradingView's "Pro-Gefühl" kommt zu 50% vom Snapping.

Implementierung:
- Snap-Targets sammeln:
  * Alle Candles OHLC in Viewport
  * Endpoints anderer Drawings
  * Grid-Lines
  * Price-Levels (Round Numbers, Fibonacci Levels)
- Snap-Distance: ~5px in Screen Space
- Snap-Feedback: Visuell (gepunktete Linie, Highlight)
- Modifier Keys: Alt deaktiviert, Shift locked Achse

Code:
```rust
fn find_snap_target(point: Point, targets: &[SnapTarget]) -> Option<Point> {
    targets.iter()
        .filter(|t| t.screen_distance(point) < SNAP_THRESHOLD)
        .min_by_key(|t| t.screen_distance(point))
        .map(|t| t.point)
}
```

Aufwand: 1-2 Wochen für Basic, 4-6 Wochen für TradingView-Level

Problem D: RENDERING PERFORMANCE
- 100 Trendlines = 100 Linien + 200 Handles
- Pro Frame:
  * Transform Coordinates (Data → Screen)
  * Culling (out-of-viewport)
  * Draw Lines
  * Draw Handles (nur wenn Selected/Hovered)
- Optimierung:
  * Batch Rendering
  * Dirty Flags (nur re-render wenn changed)
  * WebGL für >500 Objekte

Loom's Adaptive FPS hilft, aber Drawings müssen ins Invalidation-System.


2. STATE MANAGEMENT (WASM ↔ JS BOUNDARY)
-----------------------------------------
Problem:
- WASM hat State (Candles, Indicators, Drawings)
- JS hat State (UI, Chart Instance, Events)
- Wer ist "Source of Truth"?

Aktuell (konzept.txt + Code):
- WASM: "Truth", JS ist "Shell"
- WASM exported Funktionen
- JS ruft WASM, WASM ruft JS Callbacks (window.loomChart)

Herausforderungen:

A) LATENZ
- Jeder WASM-Call = Boundary-Cross
- JSON Serialization (serde_json) nicht gratis
- Bei 60 FPS = max 16ms Budget
- Mousemove @ 120Hz = 8ms Budget

Lösung:
- Event-Queue in JS, batch zu WASM (1x pro Frame)
- Render-Commands von WASM → JS als typed arrays

B) MEMORY
- JS GC ≠ Rust Ownership
- Wer besitzt Candle-Array?
  * Aktuell: Dupliziert (WASM + JS)
  * Besser: WASM linear memory, JS views darauf
  * Problem: wasm-bindgen unterstützt das begrenzt

C) DEBUGGING
- WASM Stack Traces schlecht
- console.log in Rust geht, aber umständlich
- Feature Flags helfen: "dump_primitives", "debug_layout"

D) VERSIONING
- WASM Module gecached (browser)
- Bei Update: Cache invalidieren
- Trading App braucht Cache-Busting

Loom's Lösung (packages/wasm-core/src/lib.rs):
```rust
thread_local! {
    static APP_STATE: RefCell<Option<Rc<RefCell<AppState>>>> = ...;
}
```
→ Sauber, kein unsicheres Global State.

Aber: JS muss Candles auch speichern (chart.setData(candles))
→ Duplikation

Besser:
WASM gibt "Commands", JS führt aus:
```rust
pub enum RenderCommand {
    SetCandles { data: Vec<u8> }, // Packed binary
    UpdateCandle { index: usize, candle: Candle },
    DrawLine { id: String, points: Vec<Point>, color: Color },
}
```


3. HIT-TESTING & INTERACTION
-----------------------------
Siehe "Drawing Tools" oben.

Zusätzlich: MULTI-LAYER Hit-Testing

Layer-Hierarchie (vom Vordergrund):
1. Handles (Priority 1)
2. Selected Drawings (Priority 2)
3. Unselected Drawings (Priority 3)
4. Candles/Indicators (Priority 4)
5. Background (Grid, etc.)

Hit-Testing-Algorithmus:
```
on MouseMove(x, y):
  for layer in [handles, selected, drawings, candles]:
    hit = layer.hitTest(x, y)
    if hit:
      setCursor(hit.cursor)
      emit HoverEvent(hit)
      return
  setCursor(default)
```

Performance:
- Spatial Index essentiell
- Cursor-Feedback: Max 5ms Latenz (sonst fühlt es sich laggy an)


4. PERFORMANCE BEI VIELEN OBJEKTEN
-----------------------------------
Szenarien:
- 10.000 Candles
- 500 Drawings
- 20 Indikatoren (each 10.000 points)
- 3 Panels

Bottlenecks:

A) COORDINATE TRANSFORMS
- Pro Frame: Jedes Objekt transformieren (Data → Screen)
- 10k Candles × 4 coords = 40k transforms
- Optimierung:
  * Viewport Culling (nur sichtbare)
  * Transform-Cache (bei statischer Viewport)

B) RENDERING
- Canvas2D: ~10k Objekte @ 60fps → grenzwertig
- WebGL: ~100k Objekte @ 60fps → easy
- Problem: Canvas2D API einfacher, WebGL komplex

Loom's Ansatz:
- Canvas2D aktuell
- WebGPU experimental
- Adaptive FPS reduziert bei Komplexität

Empfehlung:
- Canvas2D für <500 sichtbare Objekte
- WebGL für >500
- Auto-Switch basierend auf Complexity

C) INDICATOR CALCULATION
- 20 Indikatoren × 10k Candles = viel Arbeit
- Aktuell: Rust (schnell)
- Problem: Bei jedem Candle-Update → Recalc
- Optimierung:
  * Incremental Update (nur neue Candle)
  * Caching (Indicator-Results speichern)
  * Web Workers (off main thread)

WASM in Worker:
```js
const worker = new Worker('wasm-worker.js');
worker.postMessage({ cmd: 'calculate', indicator: 'rsi', candles });
worker.onmessage = (e) => chart.updateIndicator(e.data);
```

Loom hat das konzeptuell (packages/wasm-core), aber nicht implementiert.


5. UNDO/REDO SYSTEM
-------------------
TradingView ohne Undo/Redo wäre unbrauchbar.

Pattern: COMMAND
```rust
trait Command {
    fn execute(&self, state: &mut State);
    fn undo(&self, state: &mut State);
}

struct AddDrawing { drawing: Drawing }
impl Command for AddDrawing {
    fn execute(&self, state: &mut State) {
        state.drawings.push(self.drawing.clone());
    }
    fn undo(&self, state: &mut State) {
        state.drawings.pop();
    }
}
```

Stack:
```rust
struct UndoStack {
    commands: Vec<Box<dyn Command>>,
    current: usize,
}
```

Operationen:
- Do: Execute + push to stack
- Undo: current--, execute undo
- Redo: execute, current++

Herausforderung:
- Commands müssen serializierbar sein (für Persistence)
- Bei komplexen Transformationen: Wie "undo"?
  * Option A: Store Inverse Command
  * Option B: Store Full State Snapshot (Memory-hungry)

TradingView: ~100 Undo-Schritte

Loom: Noch nicht implementiert.

Aufwand: 1-2 Wochen für Basic, 4 Wochen für Production-ready


6. TEXT RENDERING & LABELS
---------------------------
Problem: Text in Canvas2D ist schwierig.

Herausforderungen:

A) TEXT MEASUREMENT
- measureText() ist langsam (DOM-Call)
- Caching essentiell
- Font-Size, Family, Weight beeinflussen

B) TEXT RENDERING
- fillText() okay, aber:
  * Aliasing bei kleinen Fonts
  * Scaling (HiDPI) → blurry
  * Rotation → schlechte Qualität

C) MULTI-LINE TEXT
- Canvas2D hat kein "Textbox"-Konzept
- Manuell: Zeilen splitten, Position berechnen

D) LABELS AUF DRAWINGS
- Trendline: "Slope: 12.5%"
- Fib: "0.618 - $45,234"
- Label-Position: Auto (avoid overlap) vs Manual
- Label-Style: Background, Border, Arrow

Lösungen:

Option A (Canvas2D + Caching)
```js
class TextCache {
  measure(text, font) {
    const key = `${text}:${font}`;
    if (!this.cache.has(key)) {
      this.cache.set(key, ctx.measureText(text));
    }
    return this.cache.get(key);
  }
}
```

Option B (DOM Overlay)
- Chart = Canvas
- Labels = DOM (absolute positioned)
- Vorteile: CSS, Interaktivität, Schärfe
- Nachteile: Z-Index-Management, Performance

Option C (SDF Text Rendering - WebGL)
- Signed Distance Fields
- Super-scharf bei Zoom
- Komplex zu implementieren

Loom: Aktuell Canvas2D Text (packages/chart-core/src/renderer/canvas.ts)

Für MVP: Option A ausreichend
Für TradingView-Level: Option B (Hybrid)

- Complexity: Zwei Module koordinieren

- Vorteile:

- Nachteile:

Alternative B: Micro-Modules

Empfehlung:

STATE MANAGEMENT KONZEPT
------------------------
Aktuell: WASM = Truth, JS = Shell

Flow (Candle Update):
1. Phoenix sends candle_update via WebSocket
2. WsClient (WASM) empfängt message
3. AppState::on_candle_update(candle)
4. AppState.push_candle(candle)
5. AppState.send_candles_to_chart()
6. bridge::update_chart() → JSON serialization
7. window.loomChart.updateCandle(json)
8. Chart (JS) parses + renders

Bewertung:
+ Klare Zuständigkeiten
+ WASM besitzt State → reproduzierbar

- JSON Serialization bei jedem Update
- State dupliziert (WASM + JS Chart hat Candles)

Optimierung:
```rust
// Statt JSON:
pub fn get_candles_binary() -> Vec<u8> {
    // Packed binary format
}
```

JS:
```js
const buffer = wasm.get_candles_binary();
const view = new Float64Array(buffer);
// Parse: [time, o, h, l, c, v, time, o, h, l, c, v, ...]
```

Aufwand: 1 Woche


EVENT-FLOW (Alpine.js → WASM → Canvas)
---------------------------------------

Beispiel: User wählt Symbol

1. Alpine.js:
   ```html
   <select x-model="selectedSymbol" @change="onSymbolChange()">
   ```

2. app.ts:
   ```js
   onSymbolChange() {
     this.fetchCandles(); // REST API
   }
   ```

3. fetchCandles():
   ```js
   const candles = await fetch('/api/candles?symbol=...');
   updateChartCandles('main-chart', candles);
   ```

4. chart-bridge.ts:
   ```js
   function updateChartCandles(id, candles) {
     const chart = getChart(id);
     chart.setData(candles);
   }
   ```

5. chart.ts:
   ```js
   setData(candles) {
     this.candleSeries.setData(candles);
     this.fitToData();
     this.needsRender = true;
   }
   ```

6. Chart Render Loop:
   ```js
   render() {
     this.renderer.beginFrame();
     this.renderCandles();
     this.renderer.endFrame();
   }
   ```

Bewertung:
+ Flow ist nachvollziehbar
+ Alpine.js → direkter API-Call (kein WASM-Umweg für diesen Fall)

- WASM wird umgangen (State-Drift-Risiko)
- trading_ui (WASM) weiß nichts von diesem Candle-Load

Problem:
Wenn später WASM Indicator-Berechnung macht, braucht WASM die Candles.
→ Candles müssen auch zu WASM

Lösung:
```js
fetchCandles() {
  const candles = await fetch(...);
  // 1. Update WASM
  await loadWasm().then(wasm => wasm.load_candles(JSON.stringify(candles)));
  // 2. Update Chart
  updateChartCandles('main-chart', candles);
}
```

Oder: WASM macht REST-Call (fetch in WASM via web-sys)

ARCHITEKTUR-ENTSCHEIDUNGEN (Wichtig)
-------------------------------------

ENTSCHEIDUNG 1: Rendering-Strategie

   Optionen:
   A) Canvas2D (aktuell)
   B) WebGL
   C) WebGPU
   D) Hybrid (Canvas2D UI, WebGL Candles)

   Empfehlung: D (Hybrid)

   Begründung:
   - Canvas2D: Gut für <500 Objekte
   - WebGL: Nötig für >500 Drawings + 10k Candles
   - WebGPU: Zu früh (Browser-Support noch gering)
   - Hybrid: Best of both (Canvas2D für Controls, WebGL für Data)

   chartcore hat bereits Canvas2D-Renderer (WASM).
   chart-core könnte WebGL-Renderer bekommen.

   Aufwand: 4-6 Wochen für WebGL-Renderer

ENTSCHEIDUNG 2: Drawing State Ownership

   Optionen:
   A) WASM besitzt alles
   B) JS besitzt alles
   C) Hybrid (WASM Truth, JS Cache)

   Empfehlung: C (Hybrid)

   Begründung:
   - WASM Truth: Undo/Redo, Persistence, Backtesting
   - JS Cache: Hit-Testing, Rendering (Performance)
   - Sync via Commands

   Trade-off:
   + Beste Performance
   + Klare Zuständigkeiten
   - Komplexität (Sync)

   Aufwand: 2 Wochen für Command-API

ENTSCHEIDUNG 3: UI Framework

   Aktuell: Alpine.js

   Optionen:
   A) Behalten (Alpine.js)
   B) Wechsel zu React
   C) Wechsel zu Solid.js
   D) Wechsel zu Svelte

   Empfehlung: B (React) ODER C (Solid.js)

   Begründung:
   - Alpine.js gut für MVP
   - Ab Phase 2 (Drawings): Component-Hierarchie wichtig
   - React: Ökosystem, Hiring, Libraries
   - Solid.js: Schneller, moderner, kleinerer Bundle

   Timing: Nach MVP, vor Phase 2

   Aufwand: 2-3 Wochen Migration

ENTSCHEIDUNG 4: Backtesting-Engine

   Wohin gehört die Logik?

   Optionen:
   A) WASM (Client-side)
   B) Phoenix (Server-side)
   C) Separater Service (Rust)

   Empfehlung: C (Separater Service)

   Begründung:
   - Backtesting ist compute-heavy
   - Multi-Threading nötig (Rust tokio/rayon)
   - WASM single-threaded (WebWorkers helfen, aber limitiert)
   - Phoenix gut für API, aber Elixir nicht optimal für Numerik
   - Separater Rust Service: perfekt

   Architektur:
   ```
   Frontend → Phoenix API → Backtesting Service (Rust/gRPC)
                                   ↓
                              Supabase (Results)
   ```

   Aufwand: 4-6 Wochen
