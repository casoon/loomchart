2) Datenmodell und DB Schema (Supabase Postgres)

Definiere Tabellen (SQL DDL) + Best Practices.

Tabelle candles
	•	Spalten:
	•	source (text) – z.B. “capitalcom”
	•	symbol (text)
	•	timeframe (text) – “1m”, “5m”, …
	•	ts (timestamptz) – Startzeit des Candle-Buckets
	•	o, h, l, c (numeric)
	•	v (numeric)
	•	is_final (boolean) – Candle abgeschlossen
	•	updated_at (timestamptz)
	•	Constraints:
	•	UNIQUE (source, symbol, timeframe, ts)
	•	Indizes:
	•	(source, symbol, timeframe, ts DESC)
	•	Upsert-Pattern:
	•	laufende Candle: same ts updaten, is_final=false
	•	bei Close: set is_final=true

Tabelle indicator_definitions
	•	Speichert welche Indikatoren existieren, versioniert
	•	Spalten:
	•	id (uuid)
	•	name (text) – “rsi”, “ema”, …
	•	version (int)
	•	params_schema (jsonb) – Beschreibung/Defaults
	•	description (text)

Tabelle indicator_instances
	•	Konkrete aktivierte Indikatoren pro Chart/User/Watchlist
	•	Spalten:
	•	id (uuid)
	•	symbol, timeframe, source
	•	definition_id (uuid)
	•	params (jsonb)
	•	created_at

Tabelle indicator_values
	•	Ergebniszeitreihe
	•	Spalten:
	•	instance_id (uuid)
	•	ts (timestamptz)
	•	value (numeric) oder values (jsonb) für multi-line (MACD)
	•	UNIQUE (instance_id, ts)
	•	Index (instance_id, ts DESC)

Best Practice Hinweise
	•	Candle-Bucket ts immer auf Timeframe-Start runden (Server-seitig deterministisch).
	•	“Final candle” ist wichtig fürs Backtesting/Signals.
	•	Schreibe SQL für “last N candles” effizient (Limit + order).

3) Phoenix Backend Design (Fly.io)

Gib eine modulare Phoenix-Struktur vor:

Kontext-Module
	•	MarketData Context:
	•	Query functions: list_candles/4, get_latest_candle/3
	•	Insert/upsert: upsert_candle/1
	•	Indicators Context:
	•	registry/definitions
	•	compute pipeline (siehe Abschnitt 6)
	•	Streaming:
	•	Channel Topics: "candles:{source}:{symbol}:{tf}"
	•	Broadcast: Phoenix.PubSub + Endpoint.broadcast

Channels
	•	CandlesChannel:
	•	join topic
	•	beim join optional: last candle + server time senden
	•	events:
	•	candle_snapshot (initial N candles optional)
	•	candle_update (laufende Candle Update)
	•	candle_final (Candle geschlossen)
	•	optional: indicator_update (wenn Server Indikatoren streamt)

REST API
	•	GET /api/candles?source=&symbol=&tf=&from=&to=&limit=
	•	GET /api/indicators/definitions
	•	POST /api/indicators/instances (später)
	•	GET /api/indicators/values?instance_id=&from=&to=&limit=

Hosting / Fly.io Best Practices
	•	Stateless App, Supabase Postgres als externes DB
	•	Konfiguration via ENV: DATABASE_URL, SECRET_KEY_BASE, etc.
	•	Healthcheck endpoint
	•	WebSocket Zeitouts beachten, Reconnect-freundliche Topics

6) Indikatoren: modular, erweiterbar, Best Practice

Ziel: Indikatoren sollen leicht nachrüstbar sein, ohne Chaos.

Zwei Modi (planen, aber MVP simpel halten)
	1.	Client-computed (WASM) – schnell fürs UI
	•	WASM berechnet z.B. EMA/RSI aus Candle Stream
	•	Vorteil: keine Serverlast, sehr responsive
	•	Nachteil: Ergebnisse nicht zentral gespeichert
	2.	Server-computed (Phoenix Worker) – “source of truth”
	•	Server berechnet Indikatoren und speichert in indicator_values
	•	streamed optional Updates via Channel
	•	Vorteil: reproduzierbar, für Backtesting/Signals nutzbar

Empfehlung: MVP client-computed, später server-computed ergänzen.

Indikator-Plugin Pattern (WASM und/oder Server)

Definiere ein Interface:
	•	Input: stream von Candles
	•	Output: stream von Points (ts, value) oder multi-values

Beispiel (konzeptionell):
	•	Indicator trait (Rust) / behaviour (Elixir)
	•	init(params, tf)
	•	on_candle(candle) -> [points]
	•	reset(history)

Best Practices für Indikatoren
	•	Rolling Window effizient: Ringbuffer/Deque
	•	“Final candle”: bei is_final=true committen, sonst nur “preview” updaten
	•	Numerische Stabilität: keine unnötigen float Fehler; klare rounding policy
	•	Multi-line Indikatoren (MACD): ein values: {macd, signal, hist} Schema

7) Rust→WASM UI Core: Responsibilities + API

Beschreibe die WASM Export API und State Machine:

Exports (WASM)
	•	init(config_json)
	•	connect(source, symbol, tf, ws_url)
	•	disconnect()
	•	set_symbol(symbol) / set_timeframe(tf) (macht intern resync)
	•	toggle_indicator(name, params_json, enabled)

Internes Verhalten
	•	beim connect:
	1.	REST history laden (last N candles)
	2.	chart.setCandles(…)
	3.	WS subscribe
	•	bei candle_update:
	•	update state
	•	chart.updateCandle(…)
	•	indikator pipeline: update line points
	•	bei reconnect:
	•	exponential backoff
	•	REST resync ab letztem final candle
